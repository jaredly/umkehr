# Research: Self-Contained Jigsaw Canvas

## Goal

Update `examples/react-crdt` so the Jigsaw board is self-contained: puzzle pieces should no longer render outside the board area and overlap surrounding UI. The board should behave like a bounded canvas with pan/zoom/scroll-style navigation and a minimap.

The most natural fit in this codebase is a viewport plus transformed canvas, similar to the whiteboard app, while keeping the existing per-piece canvas rendering for masks and image slices.

`JigsawPanel` currently renders:

- A header with title, progress, reshuffle, undo, and redo.
- A `.jigsawStage` `div` sized to the source image aspect ratio.
- A faded solved image canvas absolutely positioned inside `.jigsawStage`.
- One absolutely positioned `<button class="jigsawPiece">` per piece.
- A `<canvas>` inside each piece button, generated by `PieceCanvas`.

The coordinate system is the source image coordinate system from `currentJigsawBoard().imageSize`, currently `720 x 540`.

Pieces store and calculate positions as centers in board coordinates. The rendered CSS converts the top-left piece bounds to percentages:

```ts
'--piece-left': position.x - pieceSize.width / 2,
'--piece-top': position.y - pieceSize.height / 2,
'--piece-width': pieceSize.width,
'--piece-height': pieceSize.height,
```

The overflow bug is directly caused by `.jigsawStage`:

```css
.jigsawStage {
    position: relative;
    ...
    margin: 44px auto;
    overflow: visible;
}
```

Unplaced pieces are deliberately arranged around the border by `arrangeUnplacedPieces(board, unplaced, board.imageSize, seed)`. That function uses a large margin derived from the piece size, so piece centers can sit outside the `720 x 540` image bounds. Because `.jigsawStage` has `overflow: visible`, those pieces can visually escape the stage and cover the header, buttons, adjacent panels, sync controls, or other app chrome.

## Existing Local Pattern To Reuse

The whiteboard app already has almost the exact viewport mechanics needed:

- `WhiteboardPanel.tsx` renders a clipped `.whiteboardViewport`.
- `.whiteboardViewport` has `overflow: hidden`, `touch-action: none`, and a fixed interactive region.
- `.whiteboardCanvas` is an absolutely positioned child with `transform-origin: 0 0`.
- `useWhiteboardGestures.ts` stores local viewport state `{panX, panY, zoom}`.
- It handles:
  - pan drag through a `pan` drag state,
  - wheel zoom around the cursor,
  - zoom buttons,
  - viewport size tracking with `ResizeObserver`,
  - minimap drag/click recentering,
  - a `viewRect` for the minimap.
- `Minimap.tsx` renders an SVG preview plus a viewport rectangle.

The jigsaw work should mirror this shape, but jigsaw does not need to adopt the full whiteboard tool system.

## Recommended Direction

Keep the existing DOM structure for individual pieces, but put it inside a clipped viewport/canvas pair:

```tsx
<div className="jigsawViewport" ...>
    <div
        className="jigsawCanvas"
        style={{
            width: boardSpace.width,
            height: boardSpace.height,
            transform: `translate(${viewport.panX}px, ${viewport.panY}px) scale(${viewport.zoom})`,
        }}
    >
        <SolvedImageCanvas ... />
        {pieces...}
    </div>
    <JigsawMinimap ... />
</div>
```

This is closer to the existing whiteboard architecture than a full single `<canvas>` renderer. It also preserves:

- keyboard and screen-reader addressability of each piece button,
- existing pointer handling on piece buttons,
- existing `PieceCanvas` mask/image rendering,
- existing z-index ordering,
- existing tests around layout and snap logic.

The term "SVG canvas" in the task can be satisfied by the minimap SVG and by making the main area a bounded canvas-like viewport. A full SVG-only implementation would require converting the canvas clipping/masking code to SVG masks or image patterns. That is more work and gives little benefit for the current rectangular generated pieces.

## Board Space

The current source image bounds are too small to hold the intended unplaced-piece ring. If the viewport clips strictly to `720 x 540`, the reshuffled unplaced pieces that intentionally sit around the border will be partly hidden.

Recommended approach:

- Introduce a larger logical board space for interaction, separate from `board.imageSize`.
- Keep the solved image at an offset inside that space.
- Arrange unplaced pieces around the solved image inside the larger board space.

For example:

```ts
const pieceSize = estimatedPieceSize(board);
const boardPadding = Math.max(pieceSize.width, pieceSize.height) * 1.4;
const boardSpace = {
    width: board.imageSize.width + boardPadding * 2,
    height: board.imageSize.height + boardPadding * 2,
};
const imageOffset = {x: boardPadding, y: boardPadding};
```

Then convert rendered positions by adding `imageOffset`:

- CRDT positions remain in image coordinates.
- Snap/layout math remains unchanged.
- Rendering uses `renderedPosition + imageOffset`.
- Pointer coordinates from screen should subtract `imageOffset` before feeding existing drag/snap logic.
- The solved image is positioned at `imageOffset`.
- The minimap uses board-space positions, including the image outline and pieces.

This avoids migrating stored state and keeps the puzzle model about the image rather than about viewport padding.

## Gesture Changes

Add jigsaw-local viewport state and helpers, probably in `JigsawPanel.tsx` first unless it grows large enough for `useJigsawViewport.ts`.

Needed state:

```ts
type Viewport = {
    panX: number;
    panY: number;
    zoom: number;
};
```

Recommended constants:

- `minZoom = 0.35`
- `maxZoom = 2.5`
- `wheelZoomInFactor = 1.08`
- `wheelZoomOutFactor = 0.92`
- minimap size similar to whiteboard: `120 x 90` or proportional to board space.

Pointer conversion should switch from current stage-percentage math:

```ts
x: ((event.clientX - rect.left) / rect.width) * board.imageSize.width
```

to viewport-transform math:

```ts
const boardX = (event.clientX - rect.left - viewport.panX) / viewport.zoom;
const boardY = (event.clientY - rect.top - viewport.panY) / viewport.zoom;
return {
    x: boardX - imageOffset.x,
    y: boardY - imageOffset.y,
};
```

Piece drag can stay in image coordinates. Dragging a connected component should still use the same `buildPuzzleLayout`, `snapCandidates`, `positionPatch`, and `connectionPatch` logic.

Panning options:

- Wheel with Ctrl/Cmd zooms around cursor, matching whiteboard.
- Drag empty viewport to pan.
- While dragging a piece, pointer events should continue moving the piece, not pan.
- The main viewport can use `overflow: hidden`; browser scrollbars are not necessary if pan/zoom and minimap exist.

For read-only history previews, pan/zoom should remain enabled, while piece drag and state-changing actions stay disabled.

## Rendering Changes

Replace `.jigsawStage` with at least two layers:

- `.jigsawViewport`: clipped panel region, owns wheel/pan/minimap events.
- `.jigsawCanvas`: transformed logical board space.

Update CSS:

- `.jigsawViewport` should have a stable height like `min(70vh, 720px)` with a reasonable `min-height`.
- It should use `overflow: hidden`, `position: relative`, `touch-action: none`, and `user-select: none`.
- `.jigsawCanvas` should be `position: absolute`, `left: 0`, `top: 0`, `transform-origin: 0 0`.
- `.jigsawSolvedImage` should no longer rely only on `inset: 0`; it should be positioned with `left/top/width/height` in board-space pixels.
- `.jigsawPiece` should switch from percentage positioning to pixel positioning in board space, e.g. `left: var(--piece-left-px)` and `top: var(--piece-top-px)`, because the transformed canvas now has fixed logical dimensions.

Pixel positioning is simpler and less error-prone once the viewport transform controls scaling.

## Minimap

Add a jigsaw-specific minimap component rather than over-generalizing the whiteboard one immediately.

It should render:

- The full logical board-space background.
- A faint solved-image rectangle at `imageOffset`.
- One rectangle per visible piece, using rendered positions and `pieceSize`.
- Optional stronger fill/stroke for placed/active pieces.
- A viewport rectangle based on the current pan/zoom and viewport element size.

Interactions:

- Pointer down/click recenters the viewport on the minimap point.
- Pointer drag updates recenter continuously.
- Use pointer capture like `Whiteboard Minimap`.

Because the board-space aspect ratio may differ from `720 / 540` once padding is added, compute minimap scale from both width and height:

```ts
const scale = Math.min(minimapInnerWidth / boardSpace.width, minimapInnerHeight / boardSpace.height);
```

Then offset the drawing inside the minimap if letterboxed.

## Risks

- If pointer conversion mixes board-space and image-space coordinates, dragging and snapping will feel offset. Keep model positions in image coordinates and apply `imageOffset` only at the render/input boundary.
- Window-level pointer listeners may be easier than relying only on viewport pointer events once the viewport clips content. The whiteboard already uses window listeners for drags, which is a good reference.
- Dragging a piece near the clipped viewport edge will not reveal offscreen space unless the user pans separately. Auto-pan near edges would be nice but is not required by the task.
- Minimap rectangles for pieces are approximate if future piece masks become non-rectangular. That is acceptable for navigation.
- Browser wheel events should only intercept Ctrl/Cmd zoom to avoid preventing normal page scrolling unexpectedly. If the viewport is intended to consume all wheel events for pan/zoom, that should be a deliberate UX decision.

## Open Questions

1. Should jigsaw use a pan tool/button like whiteboard, or should empty-space drag always pan?
    - empty space drag should pan
2. Should normal wheel scroll pan the viewport, with Ctrl/Cmd wheel zooming, or should wheel always zoom inside the jigsaw viewport?
    - optimize for touchpad. wheel scroll should pan
3. Should viewport pan/zoom be local UI state only, or should it sync through ephemeral presence so collaborators can see another user navigating?
    - local UI only
4. Should the board support auto-pan while dragging a piece near the viewport edge?
    - no
5. Should read-only/history panels allow reshuffle of local unplaced-piece layout? Today `Reshuffle` is not disabled by `readOnly` because it only changes local layout state.
    - it's fine to be enabled
6. Is it acceptable to keep each piece as an HTML button containing a small canvas, or is a single SVG/canvas-rendered scene specifically required?
    - whatever you think would be the most performant. We're going to need to scale up to many hundred pieces
7. Should the minimap show only the authoritative CRDT positions, or should it also reflect local unplaced-piece layout and in-progress drag positions? For usability, it should reflect the same `renderedPositions` as the main board.
    - it can just do authoritative positions
