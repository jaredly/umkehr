I want our sync layer to be able to report "status issues/notices" with individual (arbitrary) pieces of state. The UI layer can decide to subscribe to these, or not, as they see fit. But this would allow the sync layer to communicate "which parts of state would be impacted by this local re-play", and the UI could highlight things according to that, so it's clearer to the user what is being impacted. Does that make sense? Can you make a .tasks/statuses/plan.md outlining what the API would look like, and any open questions?
I'm imagining something parallel to `useValue`, it would be like `useNotices` or `useStatuses` or something. And a 'status' could be "conflict" or "changed" or things like that (maybe also 'deleted in peer'). And then our Todo example would maybe draw an `outline` around todo items that contain statuses. `useStatuses` would take an optional second parameter to subscribe to 'any statuses in child paths', where the default would be only statuses on that exact path.
There might be other important uses for this kind of api that I haven't thought of yet.
