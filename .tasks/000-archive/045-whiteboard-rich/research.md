# Whiteboard rich sticky-note research

## Summary

Sticky notes currently store contents as plain `text: string` and render a `<textarea>`.
The new rich-text/peritext stack is already available in the CRDT runtime:
`createSyncedContext().useRichText(node)` returns a `RichTextBinding`, and
`RichTextEditor` renders/edits that binding through peritext-backed `richText`
patches.

The core change is likely:

- Change `StickyNoteElement.text` from `string` to `RichCollaborativeText`.
- Initialize new notes with `richText()`.
- Render notes with `editor.useRichText(editor.$.elements[id].text)` and
  `<RichTextEditor {...binding} />`.
- Update archive/minimap/preview/test helpers to use the materialized plain text
  where they currently assume a string.

The main risk is not the rich-text editor itself. The main risk is that whiteboard
is currently registered for both CRDT-backed modes and the plain history "solo"
mode, while rich-text patches intentionally require CRDT history.

## Current sticky-note path

Relevant files:

- `examples/react-crdt/src/apps/whiteboard/schema.ts`
  - `StickyNoteElement` has `text: string`.
  - `WhiteboardState.elements` is a record of tagged union elements.
  - `whiteboardSchema` is generated by typia from this shape.
- `examples/react-crdt/src/apps/whiteboard/useWhiteboardCommands.ts`
  - `addNote()` creates `text: ''`.
- `examples/react-crdt/src/apps/whiteboard/ElementViews.tsx`
  - `NoteView` keeps a local textarea draft.
  - It commits `replace` to `elements[id].text` on blur.
  - Enter without Shift blurs and commits; Escape reverts the local draft.
- `examples/react-crdt/src/apps/whiteboard/ArchiveTray.tsx`
  - Uses `element.text.trim() || 'note'` for archived note labels.
- `examples/react-crdt/tests/helpers/whiteboard.ts`
  - Test helpers locate `textarea`, fill by placeholder, and assert textarea value.
- `examples/react-crdt/src/style.css`
  - Note styling is scoped to `.whiteboardNote textarea`.

Plain string note editing means concurrent edits are last-writer-wins at the whole
field level. Replacing the field with a rich-text sentinel makes note characters
merge through peritext operations instead.

## Existing rich-text path

Relevant files:

- `src/richtext/index.ts`
  - Exports `RichCollaborativeText`, `richText()`,
    `richTextFromPlainText()`, and `materializeRichText()`.
  - The type carries typia JSON schema plugin markers:
    `x-umkehr-crdt: rich-text` and version `1`.
- `src/crdt/metadata.ts`
  - `buildMeta()` creates `kind: 'richText'` metadata whenever the schema has the
    rich-text marker.
  - The user-visible state remains the sentinel `{kind: 'rich-text', version: 1}`.
- `src/helper.ts`
  - `$text.insert/delete/mark/unmark/replace` emits `op: 'richText'` draft
    patches.
- `src/react-crdt/react-crdt.tsx`
  - `useRichText(node)` materializes CRDT metadata with
    `materializeRichTextState(meta)`.
  - Returned commands call the node's `$text` helper methods.
- `src/react-rich-text/RichTextEditor.tsx`
  - Renders a `contentEditable` textbox from `view.spans`.
  - Normal typing/deleting uses insert/delete commands.
  - Paste can preserve compatible HTML marks.
  - Toolbar supports strong, em, code, and link.

The whiteboard implementation can use the same path as the existing
`src/react-crdt/react-crdt.test.tsx` rich-text editor tests:

```tsx
const body = ctx.useRichText(ctx.$.body);
return <RichTextEditor {...body} />;
```

For whiteboard notes the equivalent binding would be:

```tsx
const text = editor.useRichText(editor.$.elements[element.id].text);
return <RichTextEditor {...text} ariaLabel="Note" />;
```

## Type/API integration concerns

`WhiteboardEditorContext` is currently `AppEditorContext`, and
`AppEditorContext` does not include `useRichText`. The CRDT runtime type
`SyncedContext` does include it, but `examples/react-crdt/src/lib/crdtApp.ts`
does not expose that method through `AppEditorContext` or `CrdtEditorContext`.

Options:

- Add `useRichText` to `AppEditorContext`.
  - This is simplest for whiteboard rendering.
  - It is inaccurate for history-backed editors because `HistoryEditorContext`
    cannot support rich-text CRDT patches.
- Add `useRichText` only to `CrdtEditorContext` and make whiteboard render only
  with a CRDT editor.
  - This matches the implementation reality.
  - It requires changing the app abstraction because `AppDefinition.renderPanel`
    currently receives only `AppEditorContext`.
- Keep the app abstraction unchanged and add a rich-text adapter/fallback for
  history mode.
  - This is more work and may undermine the goal of using peritext CRDT for note
    contents.

Important: `src/history/history.ts` explicitly throws
`Rich text patches require CRDT history.` when plain history dispatch receives a
`richText` patch. So a textarea-like fallback in solo mode would need to replace
the whole rich-text sentinel or operate on a separate plain field, neither of
which is really peritext-backed editing.

## Schema and document compatibility

Changing `StickyNoteElement.text` from `string` to `RichCollaborativeText` changes
the JSON schema and CRDT metadata shape for every note text field:

- Existing documents have primitive metadata at `elements[id].text`.
- New documents will have `richText` metadata at that path.
- Old pending `replace` updates targeting `text` as a string are not compatible
  with the new rich-text schema.
- New `richText` updates require the target path to be a rich-text field.

Whiteboard currently has:

- `WHITEBOARD_DOC_ID = 'umkehr-react-crdt-whiteboard-v3'`.
- `whiteboardApp.schemaVersion = 1`.
- No `serverSchemaConfig` migration registered in `appRegistry.ts`.
- Seed generation hardcodes `WHITEBOARD_SCHEMA_VERSION = 1`.

The implementation should choose one of:

- Treat this as a new incompatible demo document line:
  - Bump `WHITEBOARD_DOC_ID`.
  - Bump whiteboard schema version.
  - Update seed fixtures to generate rich-text note contents from scratch.
  - No migration from existing whiteboard docs.
- Add a real migration:
  - Convert each existing note `text: string` to `richText()`.
  - Rebuild CRDT metadata so `text` becomes richText metadata.
  - Import old plain text using rich-text replace/insert operations or migration
    helpers.
  - Register server/local-first migration config if those modes must preserve
    existing docs.

Given this is an example app and the current whiteboard doc id already includes
`v3`, the low-risk path is probably to bump the doc id/schema and regenerate
fixtures unless preserving local/server whiteboard documents is required.

## Seed fixture impact

`examples/react-crdt/src/lib/seed/generate.ts` edits whiteboard note text with
plain `replace` operations in several places:

- `whiteboardElementEditing()`
- `whiteboardConflictingElementEdits()`
- `whiteboardManyEvents()`
- `noteElement(...)` fixture construction

After changing note text to rich text:

- Fixture element construction should use `richText()`.
- Initial text content needs to be inserted as rich-text operations, not stored
  as a string.
- Existing `replaceWhiteboardElementField(..., 'text', '...')` calls need a
  rich-text helper that dispatches `$text.replace(richTextFromPlainText(...))`
  or emits equivalent `richText` updates.

If fixture generation only creates state snapshots and then appends normal
commands, confirm whether initial fixture text is allowed to be encoded directly
as rich-text metadata by `createCrdtDocument()`. The sentinel alone does not
contain visible text; visible text lives in CRDT metadata.

## UI impact

`RichTextEditor` renders a bare contenteditable plus a fixed-position toolbar.
The note UI will need CSS to replace textarea styling:

- Add a class wrapper or style the contenteditable via `.whiteboardNote [contenteditable]`.
- Keep `onPointerDown={(event) => event.stopPropagation()}` behavior around the
  editor root so typing/selecting text does not start note dragging.
- Keep the `.whiteboardNoteHandle` as the drag affordance.
- Decide whether Enter should insert a newline, blur/commit like today, or use a
  modifier. Rich text edits commit immediately as CRDT commands, so the old
  draft/commit model no longer applies.
- Decide how Escape should behave. There is no local draft to revert unless the
  app maps Escape to CRDT undo or blur only.
- Read-only mode needs support. `RichTextEditor` currently always renders
  `contentEditable`; it may need a `readOnly` prop or whiteboard should render a
  static `RichTextSpanView` view when read-only.

The toolbar is fixed to the viewport. That should work inside the whiteboard
viewport, but confirm it is not clipped by panel overflow and that z-index is
above note layers.

## Testing impact

Update existing Playwright helpers:

- `addNote()` should target the rich-text textbox, likely by
  `getByRole('textbox', {name: 'Note'})`.
- `expectNoteText()` should read note textbox `textContent`, not textarea value.
- Tests that press Enter to commit may need to blur another way or assert after
  typing because rich-text commands dispatch immediately.

Recommended focused tests:

- Add a note and type plain text; it appears locally.
- In local-sync mode, typed note text appears in the other panel without a blur
  commit.
- Concurrent text edits within the same note merge as rich text instead of
  replacing the whole field.
- Bold/italic/code/link formatting works inside a whiteboard note.
- Undo/redo works for note text edits in CRDT modes.
- Dragging by the handle still moves notes, while pointer down inside the editor
  does not start a move.
- Archive tray labels render materialized plain text.
- Existing smoke tests pass after locator changes.

## Open questions

- Should whiteboard continue to support solo/plain-history mode after sticky-note
  contents become rich-text CRDT fields?
- If yes, should solo mode be removed for whiteboard, changed to CRDT-only local
  history, or given a non-CRDT fallback editor?
- Do we need to migrate existing whiteboard documents, or can we bump
  `WHITEBOARD_DOC_ID` and schema version and start fresh?
- Should whiteboard register a server/local-first migration config for this
  schema change?
- What should Enter do in a sticky note: insert a newline, blur, or keep the old
  "Enter commits / Shift+Enter newline" behavior?
- What should Escape do now that edits are immediate CRDT operations?
- Should `RichTextEditor` grow a first-class `readOnly` prop for whiteboard and
  other consumers?
- Should archived-note labels and minimap/preview text use a shared
  `richTextToPlainText` helper, or should whiteboard materialize directly from
  CRDT metadata where needed?
- Should note text support the full current toolbar (`strong`, `em`, `code`,
  `link`) or only bold/italic for the first whiteboard integration?
